# m-agent 产品与架构规划

## 1. 项目定位

`m-agent` 不应被定位为“多 Agent 聊天工具”或 “AutoGen 的 TUI 包装器”，更合适的定位是：

**一个面向本地知识工作与软件交付的多智能体协作工作台。**

这个定位意味着项目的核心价值不在于让多个 Agent 轮流发言，而在于：

- 通过角色分工完成复杂任务协作。
- 让协作过程持续沉淀为工作产物。
- 提供可控的流程编排和可追溯的运行记录。
- 支撑后续评估、复盘和质量改进。

## 2. 现状判断

从当前代码实现看，项目已经具备以下基础能力：

- 配置模型：`AppConfig`、`WorkflowConfig`、`AgentConfig`、`ModelConfig`
- 模板库：多种内置协作模板
- TUI：Agent 管理、任务输入、运行展示、文件树浏览
- 编排层：AutoGen 真实模式与 Mock 回退模式
- 工作空间工具：Agent 独立目录的读写能力
- 运行落盘：任务、transcript、summary、activity 持久化

因此，当前项目不是纯概念验证，而是一个**可运行的本地多智能体协作原型**。

建议将当前成熟度定义为：

- 产品阶段：原型验证后期
- 架构阶段：可演进内核初步形成
- 工程阶段：核心功能已闭环，但平台能力尚未形成

## 3. 当前最重要的问题

### 3.1 协作协议仍然偏弱

目前的工作流主要依赖 `round_robin`、`selector` 和 prompt 自觉，缺少显式的协作规则，例如：

- 谁先开始
- 谁依赖谁的产出
- 何时允许进入下一步
- 哪些产物是必须项
- 哪些条件触发 review、重试、回退或终止

这会让多智能体协作停留在“多人轮流对话”层面，难以稳定支撑真实交付。

### 3.2 配置能力和运行能力不完全对齐

配置模型已经包含：

- `litellm`
- `max_tokens`
- `extra`

但当前执行层尚未完整消化这些字段，后续会造成“用户可配置但运行不生效”的问题。

### 3.3 工具体系过薄

当前工具主要围绕工作目录文件读写，适合演示型任务，但不够支持专业 AI 软件设计与交付场景。

后续至少要扩展：

- 仓库读取与搜索
- 配置和日志读取
- 受控命令执行
- 结构化产物写入
- 质量门禁与审核状态工具

### 3.4 缺少显式的运行状态机

目前 run 更像“一次串行执行过程”，建议引入标准状态：

- `created`
- `planning`
- `executing`
- `reviewing`
- `blocked`
- `completed`
- `failed`
- `cancelled`

这样才能支持恢复执行、人工干预、失败分析和 UI 状态可视化。

### 3.5 缺少评估体系

虽然已有 `transcript.jsonl` 和 `summary.md`，但目前更像日志，还不足以支撑：

- 哪个 workflow 更稳
- 哪个 Agent 更有效
- 哪个模型更适合当前模板
- 哪种终止原因最常见
- 哪些产物经常缺失

### 3.6 测试覆盖接近空白

当前项目没有形成有效测试层，后续一旦编排、配置和文件产物逻辑持续演进，回归风险会很高。

## 4. 目标能力模型

建议把未来版本目标能力明确为以下五项：

### 4.1 任务合同化

用户输入不只是自然语言，而应转化成结构化任务规格，至少包含：

- 目标
- 输入上下文
- 约束
- 输出要求
- 验收标准

### 4.2 编排显式化

系统需要支持从“对话协作”升级到“流程协作”，包括：

- Pipeline
- 审核回路
- 经理派工
- 辩论裁决
- 单 Agent 执行

### 4.3 工具权限化

不同 Agent 应拥有不同的：

- 可见目录
- 可写目录
- 可调用工具
- 可读取上游产物范围

### 4.4 事件结构化

所有运行中的消息、工具调用、产物写入和状态变化都应该统一为结构化事件。

### 4.5 结果可评估化

每次 run 应具备可量化结果，用于追踪质量与优化方向。

## 5. 建议的系统分层

建议将项目明确拆成六层架构：

### 5.1 Experience Layer

职责：

- TUI 交互
- 任务提交
- run 可视化
- 配置编辑
- 会话浏览

### 5.2 Application Layer

职责：

- 组织应用级用例
- 驱动 run 生命周期
- 协调配置、模板、工件、诊断等服务

建议新增服务：

- `RunService`
- `ConfigService`
- `TemplateService`
- `ArtifactService`
- `DoctorService`

### 5.3 Orchestration Layer

职责：

- 管理协作流程与调度规则
- 适配不同 workflow 模式
- 输出统一事件流

建议拆分为：

- `mock`
- `autogen`
- `pipeline`
- `selector`
- `review_loop`

### 5.4 Agent Runtime Layer

职责：

- 维护 Agent 的运行态信息
- 跟踪当前任务、状态、工具权限、产物和指标

### 5.5 Tooling Layer

职责：

- 管理工作空间工具
- 管理仓库读取和搜索能力
- 管理受控执行能力
- 管理结构化产物工具

### 5.6 Persistence & Evaluation Layer

职责：

- run 事件落盘
- 产物索引
- 指标收集
- 回放能力
- 质量评估

## 6. 近期产品路线建议

建议聚焦三个高价值使用场景：

### 6.1 软件交付场景

流程示例：

- PRD
- 架构
- 实现
- 测试
- 文档

### 6.2 研究分析场景

流程示例：

- 问题拆解
- 搜集
- 分析
- 审稿
- 结论报告

### 6.3 代码评审场景

流程示例：

- 阅读
- 安全审计
- 性能审计
- 综合结论

这三个场景和当前模板体系、工作目录模型、产物沉淀能力天然匹配，适合作为产品收敛方向。

## 7. 核心设计原则

后续所有设计建议遵循以下原则：

1. **配置驱动**  
   核心行为应由配置和模板控制，而不是把流程硬编码进 UI。

2. **事件优先**  
   UI 展示、日志记录、回放和评估都基于统一事件源。

3. **工具受控**  
   工具必须有边界，不能让 Agent 获得过大的读写权限。

4. **产物优先于聊天记录**  
   真正对业务有价值的是结构化产物，不只是对话过程。

5. **允许失败，但必须可诊断**  
   Mock 回退、doctor 检查、失败原因记录应成为基础设施。

6. **面向后续平台化设计**  
   现在的实现可以轻量，但数据模型和模块边界要为后续扩展预留空间。

## 8. 架构演进结论

后续研发的主线建议固定为：

- 任务合同化
- 编排显式化
- 工具权限化
- 事件结构化
- 质量可评估化

如果这五条主线逐步落地，`m-agent` 才能从“本地 demo 工具”升级为“可持续演进的多智能体协作平台”。
