# m-agent V1 技术路线图

本文档描述未来 2-3 个月内的建议研发路线，用于把当前项目从“可运行原型”推进到“稳定可交付的多智能体协作内核”。

## 1. 路线目标

V1 阶段不追求做大而全的平台，而追求三件事：

1. 核心链路稳定可用
2. 多智能体协作具备明确流程约束
3. 运行结果可追溯、可诊断、可评估

## 2. 分阶段规划

## Phase 1：内核稳定化

### 目标

把当前 demo 风格的运行链路变成稳定原型，降低后续重构成本。

### 主要工作项

1. 建立 `RunService`
   - 将 TUI 中的 run 控制逻辑从 `app.py` 抽离
   - 统一 run 创建、执行、结束、失败处理

2. 引入统一事件模型
   - 从单纯 `AgentMessage` 升级为 `RunEvent`
   - 支持消息、状态变化、工具调用、产物写入等事件

3. 升级 `RunArtifacts`
   - transcript 之外增加事件日志和索引
   - 为回放、评估和问题定位提供基础

4. 正式接入 `doctor` 命令
   - 将已有诊断能力接入 CLI
   - 支持配置、依赖、模型认证快速检查

5. 增加基础自动化测试
   - 配置模型
   - 工作空间路径安全
   - 产物写入
   - Mock/AutoGen fallback
   - 模板实例化

### 交付物

- `RunService` 初版
- `RunEvent` 初版
- 结构化 run 落盘
- `magent-tui doctor`
- 基础测试套件

### 验收标准

- `app.py` 不再直接承担完整运行编排责任
- run 失败时可以明确定位失败阶段和失败原因
- 开发人员可通过 `doctor` 判断环境与配置问题
- CI 或本地可稳定跑通基础测试

### 风险

- 事件模型定义过早可能产生返工
- UI 和运行逻辑拆分时需要控制影响面

## Phase 2：工作流能力化

### 目标

从“轮流发言”升级为“有约束的协作流程”。

### 主要工作项

1. 新增 `pipeline` workflow
   - 适用于 PRD -> 架构 -> 实现 -> QA 这类线性流程

2. 新增 `review_loop` workflow
   - 适用于实现 -> 审查 -> 修正 -> 验收的闭环

3. 扩展 `WorkflowConfig`
   - 增加 routing rules
   - 增加 retry limit
   - 增加 review_required
   - 增加阶段依赖关系

4. 引入“必须产物”检查
   - 某些阶段没有生成目标文件时不得进入下一阶段

5. 引入“终止原因”分类
   - 正常完成
   - 触发终止词
   - 超轮次
   - 缺失产物
   - 运行异常

### 交付物

- `pipeline` 编排器
- `review_loop` 编排器
- 扩展后的 workflow 配置
- 基础门禁检查能力

### 验收标准

- `product_sprint` 和 `dev_delivery` 模板可在 pipeline 下稳定运行
- QA 未产生目标产物时 run 不应直接进入 completed
- summary 中可见明确终止原因

### 风险

- 与现有 AutoGen GroupChat 的抽象边界需要重新划分
- 流程和 prompt 的职责边界需要清晰

## Phase 3：工具体系扩展

### 目标

把当前“文件写入型 Agent”升级为“能处理真实软件任务的 Agent”。

### 主要工作项

1. 扩展仓库读取工具
   - 读取项目文件
   - 列出目录
   - 读取配置和日志

2. 扩展代码搜索工具
   - 搜索文件名
   - 搜索符号
   - 搜索关键配置项

3. 扩展受控执行工具
   - 运行测试
   - 运行 lint
   - 运行简单命令
   - 保留权限边界与审计信息

4. 引入结构化产物工具
   - 写 PRD
   - 写架构说明
   - 写测试计划
   - 写报告摘要

5. 增加工具权限模型
   - 每个 Agent 的读写范围
   - 每个 Agent 的允许工具集

### 交付物

- `tools/repo.py`
- `tools/search.py`
- `tools/shell.py`
- `tools/artifact_writer.py`
- `tool_permissions` 配置模型

### 验收标准

- 不同 Agent 可以获得不同工具能力
- 工具调用可以在事件流和 run 记录中看到
- 研发模板能完成对仓库进行读取、分析和产物输出的闭环

### 风险

- 工具一旦放开，安全边界和可控性会迅速复杂化
- shell 工具必须有严格边界，否则后续会引入风险

## Phase 4：评估与平台化

### 目标

形成可持续演进的质量闭环。

### 主要工作项

1. 建立基础 run 指标
   - 成功率
   - 平均轮次
   - 终止原因分布
   - 产物完整率

2. 建立 Agent 贡献视图
   - 产物数
   - 事件数
   - 关键节点参与度

3. 建立 workflow 效果对比
   - round_robin vs pipeline
   - 不同模型配置效果对比

4. 引入 run replay 能力
   - 回放一次协作过程
   - 定位在哪一步偏离

5. 为模板和 prompt 建立版本管理基础

### 交付物

- metrics 数据结构
- 回放机制
- 基础评估报告
- 模板版本管理约定

### 验收标准

- 可以从历史 run 中抽出统计指标
- 可以对一次失败 run 做回放和定位
- 可以比较不同模板或 workflow 的表现

### 风险

- 指标定义不清会导致后续数据价值不足
- 如果事件结构不统一，评估层会重复返工

## 3. 建议优先级

建议按照以下优先级推进：

1. `RunService` 和统一事件模型
2. `doctor` 命令接入
3. 基础测试
4. `pipeline` workflow
5. 产物门禁和终止原因分类
6. 仓库读取与搜索工具
7. 工具权限模型
8. 指标与回放

## 4. 版本里程碑建议

### M1：稳定原型

能力边界：

- 跑通基本任务
- 配置可检查
- 失败可定位
- 结构化日志可追踪

### M2：真实协作

能力边界：

- 支持 pipeline
- 支持 review loop
- 产物检查和门禁生效

### M3：软件交付工作台

能力边界：

- 支持读取项目上下文
- 支持产物化输出
- 支持基础质量统计

## 5. 资源建议

如果团队资源有限，建议最少按以下角色协作：

- 1 名主程：负责内核、编排、事件模型
- 1 名产品/设计负责人：负责 workflow、模板和产物合同
- 1 名应用工程师：负责 TUI 和配置体验

如果只有 1-2 名开发，建议严格控制范围，优先做软件交付场景，暂不扩展太多模板。

## 6. 路线总结

V1 阶段最重要的不是功能数量，而是形成一条稳定主线：

**结构化任务 -> 受控编排 -> 结构化事件 -> 产物沉淀 -> 质量评估**

只有这条主线成立，后续增加模板、工具和模型支持才不会把系统复杂度拖垮。
